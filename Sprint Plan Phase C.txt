Sprint Plan: Phase C (Contacts & Localization)
P0 (обязательно в этом спринте)
1) Единая политика валидации телефона (Livewire + API)
Что делаем

Вводим конфиг режима валидации (strict_region / e164).

Применяем один и тот же policy в:

CheckoutWizard (UI шаг контактов),

OrderController@store (серверный контроль).

Результат

Нет расхождений “на фронте прошло, на API упало”.

Оценка: 6–8 часов
Зависимости: нет
Риск: средний (регресс по текущим пользователям)

2) Убрать hardcode +993 и безопасно нормализовать номер
Что делаем

Убираем принудительное дописывание 993.

Нормализуем ввод:

00... → +...,

удаляем пробелы/скобки/дефисы,

сохраняем в одном формате (E.164-like).

Результат

Поддержка международных номеров, меньше ложных ошибок.

Оценка: 4–6 часов
Зависимости: задача #1
Риск: средний

3) UX ошибок контактов (inline + общий блок)
Что делаем

Явные короткие ошибки под полем телефона/имени.

Понятный пример формата рядом с input.

Не перегружаем текст кнопки submit длинными ошибками.

Результат

Пользователь понимает, что исправить до сабмита.

Оценка: 3–4 часа
Зависимости: #1, #2
Риск: низкий

4) Серверные reject reasons + trace correlation
Что делаем

Для отказов по контактам логируем reason code:

invalid_phone_format,

missing_name,

invalid_comment_length.

Привязываем к traceId для сквозной диагностики.

Результат

Быстрое расследование инцидентов без чтения “сырого” стека.

Оценка: 3–5 часов
Зависимости: #1
Риск: низкий

P1 (желательно в этом же спринте)
5) i18n для текстов контактов и ошибок
Что делаем

Выносим строки ошибок/подсказок в lang-файлы.

RU + fallback locale.

Результат

Централизованная локализация, проще поддержка.

Оценка: 2–3 часа
Зависимости: #3
Риск: низкий

6) Offline queue: “needs_user_action” для невалидных контактов
Что делаем

Перед sync проверяем контакты в queued order.

При невалидных данных не отправляем молча, а помечаем needs_user_action и показываем уведомление.

Результат

Нет “тихих” фейлов в оффлайн-сценариях.
