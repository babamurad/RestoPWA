# Техническое задание: Адаптация дизайна RestoPWA для десктопных устройств

## 1. Введение

### 1.1. Цель проекта
Адаптировать существующий мобильный дизайн приложения RestoPWA для корректного отображения на десктопных устройствах, сохранив при этом оптимальный пользовательский опыт на мобильных устройствах.

### 1.2. Текущая ситуация
Проект изначально разрабатывался исключительно для мобильных устройств с использованием следующих особенностей:
- Фиксированная ширина контейнера (`max-w-lg mx-auto` ≈ 512px)
- Горизонтальная прокрутка для списков (категории, рестораны)
- Фиксированная нижняя навигация
- Touch-ориентированные элементы управления
- PWA-оптимизации для мобильных устройств

### 1.3. Целевые устройства
- **Мобильные устройства**: ширина экрана < 768px (сохранение текущего дизайна)
- **Планшеты**: ширина экрана 768px - 1024px
- **Десктоп**: ширина экрана > 1024px

## 2. Анализ текущей архитектуры

### 2.1. Технологический стек
- **Frontend**: Blade templates, Tailwind CSS, Alpine.js
- **Backend**: Laravel с Livewire компонентами
- **PWA**: Service Worker, Web App Manifest

### 2.2. Структура шаблонов
- Основной layout: `resources/views/components/layouts/app.blade.php`
- Компоненты: header, bottom-nav, restaurant-card, product-card и др.
- Страницы: home, restaurants, cart, orders, profile

### 2.3. Текущие ограничения
- Все страницы используют `max-w-lg mx-auto` для ограничения ширины
- Bottom navigation фиксирована и занимает место на экране
- Горизонтальные списки неэффективны на широких экранах
- Отсутствие адаптивных breakpoint'ов в Tailwind конфигурации

## 3. Стратегия адаптации

### 3.1. Общий подход
- **Mobile-first**: сохранить текущий дизайн для мобильных, расширять для больших экранов
- **Responsive containers**: заменить фиксированную ширину на адаптивную
- **Компонентный подход**: модифицировать существующие компоненты с учетом responsive
- **Progressive enhancement**: добавлять десктопные фичи без ущерба для мобильных

### 3.2. Breakpoints
Добавить в `tailwind.config.js`:
```javascript
screens: {
  'xs': '475px',
  'sm': '640px', 
  'md': '768px',
  'lg': '1024px',
  'xl': '1280px',
  '2xl': '1536px',
}
```

## 5. Технические требования

### 5.1. Responsive breakpoints
- **xs (< 475px)**: чисто мобильный
- **sm (475px - 767px)**: крупный мобильный
- **md (768px - 1023px)**: планшет
- **lg (1024px - 1279px)**: маленький десктоп
- **xl (> 1280px)**: большой десктоп

### 5.2. Производительность
- Избегать загрузки лишних ресурсов на мобильных
- Оптимизировать изображения для разных разрешений
- Минимизировать перерисовки при изменении размера окна

### 5.3. Доступность
- Сохранить keyboard navigation
- Адаптивные размеры touch targets
- Корректное масштабирование текста

## 7. Критерии приемки

### 7.1. Функциональные требования
- [ ] Все страницы корректно отображаются на экранах шириной 320px - 1920px+
- [ ] Навигация адаптируется под размер экрана
- [ ] Все интерактивные элементы доступны и функциональны

### 7.2. Нефункциональные требования
- [ ] Время загрузки не увеличилось более чем на 10%
- [ ] Отсутствие горизонтальной прокрутки на всех размерах экрана
- [ ] Сохранение PWA функциональности

### 7.3. UX требования
- [ ] Единый визуальный стиль на всех устройствах
- [ ] Интуитивная навигация
- [ ] Оптимальное использование экранного пространства

## 8. Риски и mitigation

### 8.1. Потенциальные риски
- Увеличение размера бандла из-за дополнительных стилей
- Сложность поддержки двух разных layout'ов
- Возможные проблемы с Livewire компонентами

### 8.2. Меры по снижению рисков
- Использование CSS Grid и Flexbox для responsive
- Постепенная миграция компонентов
- Регулярное тестирование на мобильных устройствах

## 9. Оценка трудозатрат

- **Frontend разработка**: 12-15 дней
- **Дизайн**: 2-3 дня (адаптация существующих макетов)
- **Тестирование**: 3-4 дня
- **Итого**: 17-22 дня для одного разработчика

## 10. Рекомендации

### 10.1. Лучшие практики
- Использовать CSS Grid для сложных layout'ов
- Тестировать на реальных устройствах, не только в браузере
- Использовать CSS containment для производительности

### 10.2. Будущие улучшения
- Рассмотреть внедрение CSS-in-JS для компонентного подхода
- Добавить automated visual regression testing
- Исследовать возможность server-side rendering для лучшей производительности
