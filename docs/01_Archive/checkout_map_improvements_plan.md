# Профессиональный план улучшений checkout-карты

## Контекст
Текущий этап реализует выбор точки на карте, но для стабильной операционной работы и роста конверсии нужна полноценная система качества адреса: валидация, проверка доставляемости, предсказуемый UX и измеримость результата.

---

## 1. Цели и ожидаемый бизнес-эффект

### Цели
1. Увеличить конверсию в оформленный заказ на шаге checkout.
2. Снизить долю неуспешных доставок из‑за некорректного адреса.
3. Уменьшить нагрузку на поддержку и курьеров (уточняющие звонки).
4. Сделать поведение карты предсказуемым для пользователя.

### Целевые KPI (на 6–8 недель)
- Checkout conversion rate: **+5–12%**.
- Доля заказов с проблемой адреса: **-20–35%**.
- Среднее число звонков курьера для уточнения адреса: **-15–30%**.
- Ошибки отправки заказа из-за гео-данных: **<1%**.

---

## 2. Приоритеты (P0 / P1 / P2)

## P0 — критично для ближайшего релиза

### P0.1 Проверка доставляемости (zone check)
**Что делаем:**
- При каждом изменении координат проверяем, попадает ли точка в зону доставки ресторана.
- До подтверждения заказа показываем статус: «доставляем / не доставляем».
- Блокируем оформление вне зоны с понятным recovery-flow: самовывоз / другой ресторан.

**Результат:**
- Исключаем «тихие» недоставляемые заказы.

### P0.2 Reverse geocoding + confidence score
**Что делаем:**
- Преобразуем координаты в структурный адрес (улица, дом и т.д.).
- Считаем confidence score (high/medium/low).
- При low confidence — принудительно просим уточнение (ориентир/подъезд/дом).

**Результат:**
- Пользователь подтверждает не просто точку, а понятный адрес.

### P0.3 Ранняя пошаговая валидация (без отката по шагам)
**Что делаем:**
- Валидируем данные **на каждом шаге до перехода дальше**, а не только перед финальным submit.
- На шаге адреса/контактов проверяем телефон сразу при вводе и при `blur` (маска + нормализация + формат).
- Кнопка «Далее» активна только при валидном наборе полей текущего шага.
- Показываем inline-ошибки рядом с полем и короткий текст, как исправить.
- Сохраняем частично введенные данные, чтобы пользователь ничего не терял.
- Перед submit оставляем только финальную серверную страховочную проверку.
- Единый набор причин отказа (error codes) для фронта и аналитики.

**Результат:**
- Пользователь получает ошибку сразу в контексте поля и не вынужден возвращаться с шага 3 назад.

### P0.4 Базовая аналитика воронки
**Что делаем:**
- Логируем события:
  - `map_opened`
  - `pin_moved`
  - `reverse_geocode_success|failed`
  - `zone_check_pass|failed`
  - `checkout_submit_success|failed`
- Собираем baseline-метрики по воронке.

**Результат:**
- Появляется измеримость эффекта каждого изменения.

## P1 — улучшение UX и качества данных

### P1.1 UX карты: центр-пин + «Моё местоположение»
**Что делаем:**
- Переход на паттерн «фиксированный пин в центре, двигается карта».
- Кнопка геолокации с обработкой отказа в разрешении.
- Прогнозируемые loading/error/retry состояния карты.

### P1.2 Поиск адреса + подсказки + recent addresses
**Что делаем:**
- Автодополнение адреса.
- История последних адресов пользователя.
- Автовыбор наиболее вероятного варианта.

### P1.3 Нормализация адресных данных
**Что делаем:**
- Храним `address_source`, `geo_confidence`, `zone_check_snapshot`.
- Нормализуем телефон и поля адреса для downstream-процессов.

### P1.4 UX-правило: «никогда не отправлять пользователя назад из-за ошибок прошлых шагов»
**Что делаем:**
- Если ошибка относится к предыдущему шагу, показываем компактный блок исправления прямо на текущем экране (inline sheet/modal).
- Даём CTA «Исправить здесь», без полного возврата в начало checkout.
- Для телефона — быстрый инпут исправления в шаге подтверждения.

**Результат:**
- Уменьшаем фрустрацию и потери конверсии на последних шагах.

## P2 — операционные и продуктовые улучшения

### P2.1 Feedback loop от курьеров и поддержки
**Что делаем:**
- Причины «не нашли адрес» в стандартной таксономии.
- Регулярный разбор топ-проблемных адресов.

### P2.2 A/B эксперименты
**Что делаем:**
- Тестируем текущий flow vs flow с confidence/zone guard.
- Фиксируем uplift по конверсии и качеству доставки.

## 4. Техническая архитектура (минимально необходимая)

### API-контракты
- `POST /api/geo/zone-check`
  - request: `{ vendor_id, lat, lon }`
  - response: `{ in_zone, fee, eta_min, eta_max, reason }`

- `POST /api/geo/reverse`
  - request: `{ lat, lon }`
  - response: `{ address, components, confidence, provider }`

- `GET /api/geo/suggest?q=...`
  - response: `[{ label, lat, lon, components }]`

### Схема данных (order metadata)
- `address_source` (`map_pin|search|geolocate|manual_edit`)
- `geo_confidence` (`high|medium|low`)
- `zone_check_snapshot` (json)
- `geocoder_provider` (string)

---

## 5. Критерии готовности (Definition of Done)

Функциональность считается завершенной, если:
1. Пользователь видит подтверждённый адрес, а не только точку.
2. Перед submit явно отображается доставляемость + стоимость/ETA.
3. Заказ вне зоны не может быть оформлен без альтернативного сценария.
4. Все ключевые гео-события попадают в аналитику.
5. Регрессии покрыты тестами (feature + unit для доменной логики).
6. Ошибки телефона/контактов показываются до перехода на следующий шаг, без возврата пользователя с финальных шагов.

---

## 6. Риски и митигации

1. **Нестабильный geocoder / задержки API**  
   Митигация: кэш, таймауты, fallback на ручной ввод.

2. **Граничные кейсы полигонов**  
   Митигация: буфер на границе зоны, ручной override-процесс для саппорта.

3. **Падение конверсии из-за строгой валидации**  
   Митигация: постепенное включение через feature flags + A/B.

4. **Отказ в геолокации браузера**  
   Митигация: не блокирующий fallback на поиск/ручной выбор.

8. Реализовать подсказки адресов `GET /api/geo/suggest`.
9. Добавить recent addresses для авторизованных пользователей.
10. Подготовить A/B-конфигурацию и дешборд метрик.

## 8. Роли и ответственность

- **Backend:** zone-check, reverse geocode, правила валидации, метаданные заказа.
- **Frontend:** UI статусы доставляемости, confidence UX, обработка ошибок и recovery.
- **QA:** матрица сценариев (in-zone/out-of-zone, geocoder fail, offline, edge polygon).
- **Product/Analytics:** KPI baseline, A/B гипотезы, критерии принятия.

---

## 9. Минимальный релиз (MVP) на ближайший спринт

Если делать только самое важное в 1 спринт:
1. Zone-check + блокировка submit вне зоны.
2. Reverse geocode с отображением адреса.
3. Confidence = low → обязательное уточнение.
4. Базовые события аналитики.

Это даст максимальный эффект на снижение операционных проблем при умеренной сложности внедрения.
