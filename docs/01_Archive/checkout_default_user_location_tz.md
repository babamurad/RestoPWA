# ТЗ: Показывать текущую геопозицию пользователя по умолчанию при оформлении заказа

## 1. Цель
При открытии шага выбора адреса в checkout по умолчанию центрировать карту на текущей позиции пользователя (если разрешение дано), чтобы сократить время до выбора корректного адреса и повысить конверсию оформления.

---

## 2. Бизнес-обоснование

### Проблема
Сейчас пользователь часто стартует с дефолтных координат/последней точки, затем вручную перемещает карту. Это увеличивает время оформления и количество ошибок адреса.

### Ожидаемый эффект
- Быстрее первый meaningful action на карте.
- Меньше ручных перемещений карты.
- Выше доля заказов с корректной геоточкой.
- Снижение фрустрации на первом шаге checkout.

### KPI (первые 2–4 недели после релиза)
- Median time-to-first-valid-location: **-20%**.
- Доля checkout с успешной установкой адреса в шаге 1: **+5–10%**.
- Среднее число перемещений карты до подтверждения адреса: **-15%**.
- Адресные ошибки (операционные): **-10%**.

---

## 3. Scope

### In scope
1. Запрос геолокации браузера при открытии адресного шага.
2. Автоцентрирование карты на координаты пользователя при успехе.
3. Обновление маркера/координат и вызов стандартного пайплайна `setLocation`.
4. Fallback-сценарии: deny/timeout/unavailable/error.
5. Телеметрия сценариев (успех/отказ/таймаут).
6. Feature flag для безопасного поэтапного запуска.

### Out of scope
- Изменение логики тарификации/зон доставки.
- Полная переработка UI карты.
- Новые гео-провайдеры (если текущий справляется).

---

## 4. UX-спецификация

## 4.1. Базовый сценарий (разрешение уже выдано)
1. Пользователь открывает шаг адреса.
2. Карта показывает loader «Определяем ваше местоположение…» (до N секунд).
3. При успехе:
   - карта центрируется на координаты пользователя;
   - маркер ставится в центр;
   - вызывается стандартный `setLocation(lat, lon, 'geolocate')`;
   - запускается reverse geocode/zone-check (если уже интегрированы).
4. Пользователь видит актуальную точку и продолжает оформление.

## 4.2. Разрешение не выдано (prompt/deny)
- Если браузер требует согласие — показываем нативный prompt.
- Если пользователь отказал:
  - не блокируем checkout;
  - показываем unobtrusive сообщение: «Не удалось получить геопозицию. Выберите точку на карте вручную»;
  - карта остаётся на fallback-точке.

## 4.3. Timeout / error / unavailable
- При timeout или ошибке геолокации:
  - скрываем loader;
  - показываем компактную подсказку с CTA «Попробовать снова»;
  - не сбрасываем уже выбранные координаты.

## 4.4. Повторные открытия шага адреса
При повторном входе приоритет такой:
1. Явно выбранная пользователем точка в текущем checkout-состоянии.
2. Последняя успешная геолокация пользователя (с TTL).
3. Системный fallback-центр.

## 4.5. Правила, чтобы не ломать UX
- Нельзя «прыгать» картой после ручного действия пользователя.
- Если пользователь уже двинул карту/маркер, автогеолокация не должна перетирать его выбор.
- Ошибки геолокации не должны блокировать кнопку «Далее» (при наличии валидной вручную точки).

---

## 5. Функциональные требования

### FR-1. Инициализация геолокации
При открытии шага адреса система должна попытаться получить текущие координаты через Geolocation API.

### FR-2. Ограничение по времени
Ожидание геолокации ограничено configurable timeout (например, 5–8 сек).

### FR-3. Источник координат
При успешной автоустановке в payload сохраняется `address_source = geolocate`.

### FR-4. Защита от перезаписи
Если пользователь уже вручную изменил точку, автогеолокация не должна менять координаты.

### FR-5. Retry
Пользователь может повторно запросить геолокацию кнопкой «Определить местоположение».

### FR-6. Feature flag
Функция включается через флаг (`checkout.default_geolocate_enabled`).

---

## 6. Нефункциональные требования

1. **Производительность:** геолокация не должна увеличивать TTI шага больше чем на 100–150ms (асинхронный запуск).
2. **Надежность:** graceful fallback в 100% сценариев ошибки геолокации.
3. **Приватность:** никаких координат до явного действия браузерного permission flow.
4. **Наблюдаемость:** обязательные события/метрики по каждому исходу.

---

## 7. API / контракты данных

Новых backend API не требуется для самой геолокации, но нужно использовать существующий pipeline:
- `setLocation(lat, lon, source)` где `source='geolocate'`.

Рекомендуемые поля в checkout/order metadata:
- `address_source`: `geolocate | map_pin | search | manual_edit`
- `geolocate_attempted`: boolean
- `geolocate_status`: `success | denied | timeout | unavailable | error`
- `geolocate_accuracy_m`: number | null

---

## 8. Телеметрия (обязательная)

События:
1. `geolocate_requested`
2. `geolocate_success`
3. `geolocate_denied`
4. `geolocate_timeout`
5. `geolocate_error`
6. `geolocate_retry_clicked`
7. `geolocate_applied_to_map`

Минимальные параметры:
- `session_id`
- `checkout_id` (если есть)
- `accuracy_m`
- `lat_rounded`, `lon_rounded` (с privacy-ограничением по точности)
- `time_from_step_open_ms`

---

## 9. Edge cases

1. HTTPS отсутствует → Geolocation API недоступен.
2. iOS Safari задерживает/кеширует permission state.
3. Пользователь в VPN/desktop с неточной геопозицией.
4. Accuracy > допустимого порога (например > 200м):
   - отметить низкую точность;
   - предложить ручное уточнение.
5. Пользователь быстро закрыл модалку до завершения геолокации.

---

## 10. План реализации

### Этап 1 (1–2 дня)
- Добавить feature flag.
- Реализовать use-case получения координат при открытии шага.
- Подключить setLocation с source=`geolocate`.

### Этап 2 (1–2 дня)
- Добавить UX-состояния: loading, denied, timeout, retry.
- Добавить защиту от перезаписи ручного выбора.

### Этап 3 (1 день)
- Добавить телеметрию событий и базовый дешборд.

### Этап 4 (1 день)
- QA матрица + smoke/regression.
- Rollout через feature flag (10% → 50% → 100%).

---

## 11. QA-чеклист

1. Разрешение выдано заранее → карта стартует с текущей точки.
2. Пользователь нажал Deny → checkout не ломается.
3. Timeout → есть сообщение и retry.
4. После ручного перетаскивания автосценарий не перетирает точку.
5. Повторный вход в шаг использует приоритет источников корректно.
6. Все события телеметрии отправляются с нужными параметрами.

---

## 12. Критерии приемки (Acceptance Criteria)

1. При открытии шага адреса система пытается определить текущую геопозицию пользователя.
2. При успешной геолокации карта центрируется на пользователя, точка сохраняется как `source=geolocate`.
3. При ошибке/отказе пользователь может продолжить вручную без блокировок.
4. Реализована кнопка retry для геолокации.
5. События телеметрии фиксируют все исходы.
6. Функция полностью управляется feature flag.

---

## 13. Риски и митигации

1. **Низкая точность геолокации в части устройств**  
   Митигация: показывать accuracy, рекомендовать ручное уточнение.

2. **Падение UX из-за задержек permission prompt**  
   Митигация: асинхронный неблокирующий loader + timeout + fallback.

3. **Региональные ограничения/браузерные особенности**  
   Митигация: покрыть QA-матрицей Safari/Chrome/Android/iOS.

---

## 14. Итог для команды
Это изменение должно быть внедрено как **безопасное UX-улучшение по флагу**, не ломающее текущий ручной сценарий выбора точки. Основной принцип: **сначала пробуем показать точку пользователя автоматически, при любой ошибке мягко откатываемся на ручной выбор**.
