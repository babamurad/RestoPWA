# RestoPWA — аудит пробелов и план реализации

_Актуализировано: 6 апреля 2026_

---

## 1) Что проверено

Проведён экспресс-аудит текущего состояния:
- маршруты web/api;
- ключевые контроллеры и Livewire-компоненты;
- offline/PWA слой (`sw.js`, `CartService`);
- тестовая и build-инфраструктура (`composer.json`, `package.json`, `tests/*`).

---

## 2) Что уже закрыто (по сравнению с предыдущим планом)

1. Реализованы маршруты и экраны:
   - `GET /restaurants`
   - `GET /order/success/{id}`
2. Добавлены login/register/logout и защита пользовательских страниц через `auth` middleware.
3. Добавлены API-эндпоинты MVP-минимума:
   - `POST /api/v1/cart/sync`
   - `GET /api/v1/orders`, `GET /api/v1/orders/{id}` (под `auth:sanctum`)
4. Базовая offline-first архитектура присутствует:
   - Dexie (`CartService`),
   - Service Worker c fallback,
   - компоненты offline-индикации.

---

## 3) Что не хватает сейчас (gap list)

### P0 — критично перед полноценным релизом

1. **Нет воспроизводимого тестового запуска в чистом окружении**  
   `php artisan test` не запускается без `vendor/autoload.php` (зависимости не установлены в окружении).
2. **Не закрыт end-to-end сценарий “каталог → корзина → checkout → success” автотестом**  
   Есть feature/unit тесты по частям, но нет сквозного интеграционного теста пользовательского пути.
3. **Tenant-аудит API не формализован чеклистом**  
   Есть `SetTenantContext` и scope/тесты, но нужен единый аудит всех API-групп и исключений.

### P1 — важно для стабильности

4. **Нет JS/Frontend test harness для offline-корзины**  
   В `package.json` нет test-скриптов (Vitest/Jest/Playwright), значит Dexie/синхронизация не проверяются автоматически.
5. **Нет зафиксированного Lighthouse/PWA baseline**  
   Для релиза нужен документированный baseline (PWA/Performance/Best Practices/Accessibility).
6. **Нет регламентированного CI quality gate**  
   Нет описанного минимального пайплайна проверок (`pint`, `phpunit`, frontend build/test).

### P2 — желательно для техдолга

7. **Документация статуса задач расходится с фактической реализацией**  
   Старые пункты “не сделано” уже частично реализованы; нужен единый актуальный status board.
8. **Не подтверждён полный DoD по всем пунктам**  
   `docs/DoD.md` остаётся как общий чеклист, но без явной трассировки “пункт DoD → артефакт”.
