# ТЗ и пошаговый план: геолокация, карта и адреса доставки для Туркменистана

## 1. Назначение документа

Документ фиксирует профессиональное техническое задание и план реализации геолокации для RestoPWA без внесения изменений в код. Цель — дать понятный маршрут разработки, чтобы внедрять функциональность поэтапно и контролируемо.

## 2. Контекст и проблема

В проекте уже есть базовые элементы гео-функциональности:

- `GeoService` для геокодинга, reverse geocoding, проверки зон доставки и расчёта стоимости доставки.
- `AddressSelector` как Livewire-компонент выбора адреса.
- UI карты на Yandex Maps JS API в адресном модальном окне.
- Проверка зон доставки через PostGIS при production-использовании PostgreSQL.
- Checkout-flow, который требует адрес и координаты для оформления заказа.

Главная продуктовая проблема: в Туркменистане внешние карты и геокодеры могут плохо определять точные адреса. Поэтому нельзя строить оформление заказа только на строке адреса, которую возвращает Google/Yandex/Nominatim.

## 3. Ключевой принцип решения

> Координаты — основной источник истины. Текстовый адрес — вспомогательная подсказка. Ручной ориентир пользователя — обязательное уточнение для курьера.

Это означает:

1. Заказ должен опираться на `lat/lon`, выбранные через GPS, карту или подсказку.
2. Reverse geocoding не должен быть блокирующим шагом.
3. Если внешний провайдер не нашёл адрес, пользователь всё равно должен иметь возможность поставить точку вручную.
4. Для курьера нужно собирать дополнительные данные: ориентир, подъезд, этаж, квартира/офис, комментарий.
5. Зона доставки должна проверяться по координате, а не по строке адреса.

## 4. Цели

### 4.1. Бизнес-цели

- Уменьшить количество брошенных заказов из-за “адрес не найден”.
- Дать ресторану/курьеру точную точку доставки.
- Сделать UX понятным для городов со слабой адресной базой.
- Постепенно накопить собственную локальную базу адресов и ориентиров.

### 4.2. Технические цели

- Разделить понятия “координаты”, “адрес от провайдера” и “ручное уточнение пользователя”.
- Ввести fallback-стратегию геокодинга.
- Сделать выбор точки на карте основным сценарием.
- Подготовить PostGIS-зоны доставки к production-использованию.
- Добавить диагностику и логирование проблем с геокодингом.

## 5. Не входит в первый этап

Чтобы не усложнять MVP, на первом этапе не делать:

- полноценный курьерский трекинг в реальном времени;
- автоматическую маршрутизацию курьеров;
- собственный инстанс Nominatim;
- сложный ML/скоринг адресов;
- автоматическое исправление пользовательских адресов без модерации;
- интеграцию с несколькими картами на frontend одновременно.

## 6. Целевая модель адреса

В заказе и пользовательской сессии адрес должен храниться как структурированный объект:

```json
{
  "lat": 39.0886,
  "lon": 63.5593,
  "source": "gps|map_pin|suggestion|manual_geocoded",
  "provider": "yandex|google|nominatim|local|null",
  "formatted_address": "адрес от провайдера, если есть",
  "manual_address": "адрес, введённый пользователем",
  "landmark": "ориентир",
  "entrance": "подъезд",
  "floor": "этаж",
  "apartment": "квартира/офис",
  "courier_comment": "комментарий курьеру"
}
```

Минимально обязательные поля для оформления заказа:

- `lat`;
- `lon`;
- хотя бы одно текстовое уточнение: `manual_address`, `landmark` или `courier_comment`.

## 7. Пользовательский сценарий MVP

### 7.1. Выбор адреса

1. Пользователь открывает checkout.
2. Нажимает “Выбрать адрес доставки”.
3. Открывается модальное окно с картой.
4. Пользователь выбирает один из способов:
   - “Найти меня” через браузерный GPS;
   - поиск адреса через подсказки;
   - клик по карте;
   - перетаскивание метки.
5. Система получает `lat/lon`.
6. Система проверяет, входит ли точка в зону доставки выбранного ресторана.
7. Если точка в зоне доставки — пользователь заполняет ручное уточнение.
8. Пользователь подтверждает адрес.
9. Checkout показывает выбранный адрес, координаты скрыто хранятся в состоянии заказа.

### 7.2. Если адрес не найден

1. Пользователь вводит адрес.
2. Геокодер не возвращает уверенный результат.
3. Система показывает сообщение: “Адрес не найден. Поставьте точку на карте и добавьте ориентир”.
4. Пользователь ставит точку вручную.
5. Система принимает координаты и требует ручное уточнение для курьера.

### 7.3. Если GPS не работает

1. Система объясняет причину: нет HTTPS, пользователь запретил доступ, браузер не смог определить координаты.
2. Пользователь может продолжить через ручной выбор точки на карте.
3. Заказ не должен блокироваться только из-за недоступного GPS.

## 8. UX-требования

### 8.1. Обязательные тексты

В интерфейсе нужно явно объяснить:

- “Если карта не знает ваш адрес, поставьте точку вручную”.
- “Доставка проверяется по точке на карте”.
- “Добавьте ориентир для курьера: подъезд, этаж, рядом с чем находится дом”.

### 8.2. Состояния интерфейса

Модальное окно адреса должно иметь состояния:

- карта загружается;
- карта не загрузилась;
- GPS определяется;
- GPS запрещён пользователем;
- точка выбрана, идёт проверка зоны доставки;
- точка входит в зону доставки;
- точка вне зоны доставки;
- адрес найден через подсказку;
- адрес не найден, нужна ручная точка.

### 8.3. Ошибки должны быть actionable

Плохой пример:

> Не удалось определить адрес.

Хороший пример:

> Не удалось найти адрес по тексту. Поставьте точку на карте вручную и добавьте ориентир для курьера.

## 9. Провайдеры карт и геокодинга

### 9.1. Рекомендуемая стратегия

Для Туркменистана рекомендуется fallback-chain:

1. Локальная база адресов/ориентиров проекта.
2. Yandex Geocoder.
3. Nominatim/OpenStreetMap.
4. Google Geocoding API.
5. Ручная точка на карте без успешного reverse geocode.

### 9.2. Почему не только Google

Google Maps может плохо покрывать отдельные регионы Туркменистана на уровне домов и точных адресов. Поэтому Google нельзя считать единственным источником истины.

### 9.3. Почему нужна локальная база

Для доставки еды важны не только официальные адреса, но и привычные ориентиры:

- рынки;
- школы;
- больницы;
- торговые точки;
- районы;
- локальные названия мест;
- популярные жилые комплексы.

Такие данные лучше накапливать внутри продукта.

## 10. Локальная база адресов и ориентиров

### 10.1. Назначение

Локальная база должна улучшать поиск там, где внешние карты слабые, и уменьшать зависимость от внешних API.

### 10.2. Сущность `local_places`

Рекомендуемые поля:

```text
id
city
name
aliases
lat
lon
type
popularity
is_verified
created_by
updated_by
created_at
updated_at
```

Типы `type`:

- `district`;
- `street`;
- `landmark`;
- `market`;
- `school`;
- `hospital`;
- `restaurant`;
- `building`;
- `other`.

### 10.3. Алгоритм подсказок

1. Искать в `local_places` по `name` и `aliases`.
2. Если найден хороший локальный результат — показывать его выше внешних провайдеров.
3. Если локальных результатов нет — идти во внешние API.
4. Логировать неуспешные запросы пользователя.
5. Администратор периодически модерирует частые неуспешные запросы и добавляет их в `local_places`.

## 11. Зоны доставки

### 11.1. Основной принцип

Зона доставки ресторана должна быть полигоном/мультиполигоном. Проверка выполняется по точке `lat/lon`.

### 11.2. Требования к production

- PostgreSQL + PostGIS.
- `delivery_zones` в формате `MULTIPOLYGON`.
- SRID `4326`.
- Корректный порядок координат при проверке: `lon, lat`.
- Индекс GIST по геометрии.

### 11.3. Админский сценарий

Администратор/владелец ресторана должен иметь возможность:

1. открыть карту зоны доставки;
2. нарисовать полигон;
3. отредактировать точки полигона;
4. сохранить зону;
5. проверить тестовую точку;
6. увидеть предупреждение, если зона невалидна.

## 12. Безопасность и ограничения API-ключей

Для API-ключей обязательно:

- не хранить ключи в git;
- использовать `.env` и секреты окружения;
- ограничить frontend-ключ по домену;
- ограничить backend-ключ по IP/сервису, если провайдер поддерживает;
- разделить ключ для JS-карты и ключ для backend-geocoding;
- настроить лимиты и мониторинг расходов.

## 13. HTTPS и браузерная геолокация

Браузерная геолокация должна считаться доступной только при:

- HTTPS;
- `localhost` во время разработки;
- разрешении пользователя в браузере.

Если сайт открыт по обычному HTTP на телефоне или локальном IP, GPS может не работать. Это нужно учитывать в диагностике и пользовательских сообщениях.

## 14. Логирование и аналитика

Нужно логировать события:

- пользователь нажал “Найти меня”;
- GPS успешен;
- GPS запрещён;
- GPS timeout;
- геокодинг успешен;
- геокодинг не дал результата;
- пользователь поставил точку вручную;
- точка вне зоны доставки;
- карта не загрузилась;
- провайдер вернул ошибку/лимит.

Минимальные поля лога:

```text
trace_id
user_id/session_id
vendor_id
provider
query
lat
lon
status
error_code
error_message
created_at
```

## 16. Definition of Done

Функциональность считается готовой, когда:

- пользователь может оформить заказ, даже если внешний геокодер не знает адрес;
- координаты сохраняются в заказе;
- зона доставки проверяется по координате;
- пользователь видит понятные ошибки и инструкции;
- курьер получает координаты и ручной ориентир;
- ресторан может настроить зону доставки;
- API-ключи не хранятся в git;
- production работает через HTTPS;
- есть логирование неуспешных поисков адресов;
- есть план пополнения локальной базы адресов.

## 17. Приоритет ближайшей реализации

## 18. Главный риск

Главный риск — пытаться добиться идеального текстового адреса от внешнего провайдера. Для Туркменистана более надёжная стратегия: точка на карте + ручной ориентир + собственная локальная база популярных мест.
