# План внедрения админ-панели на Filament (MVP)

_Дата: 6 апреля 2026_

## Зачем это сейчас
Сейчас в проекте нет полноценной админ-панели для операционной работы.  
Без неё сложно:
- заводить рестораны/категории/товары;
- быстро менять доступность и цены;
- проверять реальные сценарии продаж и обработки заказов.

Цель: за 1 итерацию поднять рабочую админку на **Filament**, чтобы можно было наполнять каталог и тестировать продажи вживую.

---

## 1) MVP-объём админки

### 1.1 Обязательно (P0)
1. Авторизация в админ-панель.
2. CRUD для:
   - Restaurants
   - Categories
   - Products
3. Управление заказами:
   - список;
   - фильтр по статусу;
   - смена статуса (`pending -> confirmed -> preparing -> delivering -> delivered/cancelled`).
4. Привязка данных к tenant/vendor (чтобы не было утечек между ресторанами).

### 1.2 Желательно (P1)
1. Dashboard с метриками: заказы за день, выручка, средний чек.
2. Быстрые действия: “Скрыть товар”, “Изменить цену”, “Стоп-лист”.
3. Управление зонами доставки (если модель готова).

### 1.3 Потом (P2)
1. Роли и права (super-admin, vendor-admin, manager).
2. Управление push-подписками.
3. Экспорт заказов (CSV/Excel).

---

## 2) Технический план внедрения

## Этап A — Установка Filament и базовая конфигурация (0.5 дня)

- [ ] A1. Установить пакет Filament в проект.
- [ ] A2. Выполнить установщик панели и миграции.
- [ ] A3. Создать первого администратора.
- [ ] A4. Ограничить доступ к панели только авторизованным staff-пользователям.

**Результат:** панель открывается, вход работает, доступ защищён.

---

## Этап B — Ресурсы каталога (1 день)

- [ ] B1. Создать Filament Resource для Restaurant.
- [ ] B2. Создать Filament Resource для Category.
- [ ] B3. Создать Filament Resource для Product.
- [ ] B4. Добавить валидацию полей (цена, доступность, vendor_id).
- [ ] B5. Добавить фильтры и поиск по названию/статусу.

**Результат:** можно заводить товары и управлять меню без ручных SQL/сидов.

---

## Этап C — Ресурс заказов и операционная работа (1 день)

- [ ] C1. Создать/доработать Resource Order.
- [ ] C2. Таблица заказов: статус, сумма, время, ресторан, пользователь.
- [ ] C3. Action-кнопки смены статуса с логированием в `order_status_history`.
- [ ] C4. Фильтры: “новые”, “в работе”, “доставляются”, “завершённые”, “отменённые”.

**Результат:** можно проводить заказ от создания до завершения прямо из панели.

---

## Этап D — Tenant-безопасность и тест продаж (0.5–1 день)

- [ ] D1. Проверить, что в Filament-ресурсах действует ограничение по vendor.
- [ ] D2. Сценарий “тест продаж”:
  1) добавить товар,
  2) оформить заказ с фронта,
  3) изменить статус в админке,
  4) убедиться, что клиент видит изменения.
- [ ] D3. Зафиксировать найденные ошибки и завести backlog фиксов.

**Результат:** подтверждён рабочий цикл “товар -> заказ -> продажа”.

---

## 3) Практический чеклист для вашего сценария

Чтобы вы могли быстро начать “заносить товары и пробовать продажи”, минимально нужно закрыть:

- [ ] Рабочий вход в `/admin`.
- [ ] CRUD товаров и категорий.
- [ ] Переключение статуса заказа в админке.
- [ ] Проверенный end-to-end smoke:
  - добавили товар в админке;
  - увидели товар на витрине;
  - оформили заказ;
  - обработали заказ в админке;
  - дошли до `delivered`.

---

## 4) Риски и как не потерять темп

1. **Filament поставлен, но без tenant-ограничений** → риск утечки данных.  
   Решение: сразу внедрять scope/policy в каждый Resource.
2. **Каталог работает, но заказы не управляются** → продажи не тестируются.  
   Решение: Order Resource входит в P0, не откладывать.
3. **Нет тестового сценария продаж** → сложно понять, что реально готово.  
   Решение: обязательный smoke после каждого этапа.

---

## 5) Что делать сразу (первые 2 часа)

1. Установить Filament и поднять `/admin`.
2. Сделать Product/Category Resource хотя бы в базовом виде.
3. Добавить 3–5 тестовых товаров.
4. Проверить, что товары видны на клиентской витрине.

После этого уже можно начинать пробные продажи и итеративно исправлять ошибки.
