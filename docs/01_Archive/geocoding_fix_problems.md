Короткий вывод
Проблемы с картой и точкой доставки, судя по коду, идут не из одного места, а из связки карта → геокодинг → зона доставки → checkout. Самые вероятные причины:

Карта может вообще не загрузиться, если не задан YANDEX_MAPS_API_KEY / YANDEX_MAPS_JS_KEY, а UI при этом молча остаётся в состоянии “Загрузка карты...”.

Геокодинг по умолчанию настроен на Google, но карта используется Яндекс, а в .env.example ключи пустые — ручной ввод адреса и подсказки легко могут не работать.

Проверка зоны доставки почти гарантированно возвращает “вне зоны”, если у ресторана не сохранён валидный PostGIS MULTIPOLYGON.

Админка сохраняет зону как GeoJSON/строку/массив, но проверка зоны ожидает PostGIS geometry в колонке delivery_zones; явного преобразования через ST_GeomFromGeoJSON я не нашёл.

Начальная точка уже имеет координаты по умолчанию — 39.0886, 63.5593, поэтому система может считать, что точка выбрана, хотя пользователь фактически ничего не выбрал.

Что именно сейчас происходит
1. Клиентская карта зависит только от Яндекс JS API
В основном layout Яндекс-карта подключается только если есть config('services.yandex_maps.js_key'). Если ключа нет, <script> вообще не будет добавлен. 

При этом в самом селекторе адреса initMap() просто делает return, если ymaps отсутствует, без ошибки пользователю и без сброса isMapLoading. То есть пользователь может видеть бесконечную загрузку карты. 

Дополнительно в .env.example есть только пустой YANDEX_MAPS_API_KEY, отдельного YANDEX_MAPS_JS_KEY там нет, а GEO_DRIVER по умолчанию стоит google. 

Вывод: если на окружении не задан Яндекс JS key, карта в модальном окне адреса не инициализируется, но интерфейс не объясняет причину.

2. Карта Яндекс, а геокодинг по умолчанию Google
Конфиг задаёт Яндекс-ключи отдельно, но geo_driver по умолчанию равен google. 

GeoService в конструкторе действительно берёт GEO_DRIVER с дефолтом google. 

Метод geocodeAddress() использует только текущий driver, без fallback-цепочки: yandex, nominatim или google. 

При этом отдельный fallback-метод уже существует — geocodeWithFallback() — но селектор адреса его не использует. 

В AddressSelector::confirmAddress() при ручном вводе вызывается именно geocodeAddress($this->address), то есть без локальной базы и без fallback. 

Вывод: при дефолтной конфигурации ручной адрес идёт в Google. Если GOOGLE_MAPS_API_KEY пустой или Google плохо находит адреса Туркменабата, адрес не будет найден, хотя в коде уже есть более подходящая fallback-логика.

3. Подсказки адресов тоже не используют полную fallback-цепочку
suggestAddresses() сначала ищет локальные места, но если их меньше 5, идёт только в текущий driver, а не по всему списку geo_providers. 

Локальный поиск ограничен LocalPlace и простыми LIKE по name. 

Если выбран Google, Google suggestions отправляют запрос без принудительного добавления города/страны. 

А для Яндекса город Туркменабат добавляется прямо в запрос подсказок. 

Вывод: поведение подсказок зависит от GEO_DRIVER. При google пользователь может получать нерелевантные или пустые результаты, потому что запрос не локализуется на Туркменабат.

4. Начальная “точка доставки” задана заранее, хотя пользователь её не выбирал
Если в сессии нет сохранённого адреса, компонент ставит координаты по умолчанию 39.0886, 63.5593. 

В JS-карте эти же координаты используются как fallback при инициализации карты и маркера. 

В confirmAddress() наличие координат определяется просто как $this->lat && $this->lon. 

Почему это опасно: координаты есть уже до действия пользователя. Если зона доставки покрывает центр по умолчанию, можно подтвердить не ту точку. Если не покрывает — пользователь получает ошибку “точка вне зоны”, хотя он мог ещё ничего не выбирать.

5. Проверка зоны доставки требует валидную PostGIS-геометрию
Основная проверка зоны доставки делает SQL:

ST_Contains(delivery_zones, ST_SetSRID(ST_MakePoint(lon, lat), 4326))
То есть колонка delivery_zones должна быть настоящей PostGIS geometry, а не JSON-строкой. 

Если результат пустой или NULL, код приводит его к false. 

Миграции действительно создают delivery_zones как PostGIS POLYGON, а затем пересоздают как MULTIPOLYGON. 

Но сидер ресторанов вообще не задаёт delivery_zones — он создаёт рестораны из массива данных без зоны доставки. 

Админка прямо предупреждает: если зоны нет, все точки будут считаться вне зоны. 

Вывод: если ресторанам не заданы зоны доставки в PostGIS-формате, любая выбранная точка будет отклоняться как “за пределами зоны”.

6. Есть риск, что зона доставки сохраняется не в том формате
В админке поле delivery_zones принимает GeoJSON MultiPolygon и dehydrateStateUsing() возвращает либо decoded array, либо строку. 

В vendor-админке аналогично: textarea принимает GeoJSON и возвращает decoded array или строку. 

Но модель Restaurant не содержит cast для delivery_zones и не содержит mutator, который превращал бы GeoJSON в PostGIS geometry. 

Метод deliveryZones() пытается читать delivery_zones как строку JSON или массив. 

Проблема: это подходит для SQLite/text-поля, но не для PostGIS geometry. Для PostGIS нужно сохранять через ST_GeomFromGeoJSON(...) / ST_Multi(...) / ST_SetSRID(...), а читать через ST_AsGeoJSON(...). Сейчас такой явной конвертации в найденном коде нет.

7. На границе полигона точка может считаться “вне зоны”
Проверка использует ST_Contains. 

ST_Contains обычно не включает граничные точки полигона. Для доставки это часто нежелательно: пользователь ставит маркер на дороге/границе района и получает “вне зоны”. В соседнем методе поиска ресторанов используется ST_Intersects, который более терпим к границе. 

Вывод: даже при корректной зоне часть “пограничных” адресов может отклоняться.

Главная причина в одной фразе
Система сейчас смешивает Yandex JS карту, Google driver по умолчанию, неполную fallback-логику геокодинга, координаты по умолчанию как будто точка уже выбрана и PostGIS-проверку зоны, для которой зона, вероятно, не сохраняется корректно как geometry.

План решения без реализации
Этап 1. Развести состояния “центр карты” и “выбранная точка”
Ввести отдельный флаг, например hasSelectedPoint.

Координаты 39.0886, 63.5593 использовать только как центр карты, но не как адрес доставки.

confirmAddress() должен требовать именно выбранную точку: GPS, drag marker, клик по карте или выбранная подсказка.

До выбора точки не показывать “вне зоны доставки”.

Ожидаемый эффект: исчезнут ложные ошибки по дефолтной точке Туркменабата.

Этап 2. Сделать карту устойчивой к отсутствию Яндекс JS API
Если ymaps не загрузился — показать понятную ошибку: “Карта недоступна, введите адрес вручную или проверьте ключ Яндекс.Карт”.

Если YANDEX_MAPS_JS_KEY не задан — не оставлять бесконечный loader.

Добавить retry/error handler на <script> Яндекс.Карт.

В .env.example добавить явный YANDEX_MAPS_JS_KEY= или документировать, что YANDEX_MAPS_API_KEY должен подходить и для JS API.

Ожидаемый эффект: пользователь и разработчик будут понимать, что проблема в ключе/загрузке API, а не в Livewire.

Этап 3. Унифицировать провайдера карты и геокодинга
Рекомендованный вариант для текущего кода:

Для Туркменабата сделать GEO_DRIVER=yandex по умолчанию.

Использовать geocodeWithFallback() в AddressSelector::confirmAddress() вместо geocodeAddress().

Для подсказок тоже пройти по geo_providers, а не только по одному driver.

Все внешние запросы квалифицировать городом/страной, если пользователь ввёл короткий адрес.

Ожидаемый эффект: ручной ввод и подсказки перестанут зависеть от одного пустого/неподходящего ключа.

Этап 4. Починить модель хранения зон доставки
Нужно выбрать один канонический формат.

Вариант A — production/PostGIS
delivery_zones хранить только как PostGIS MULTIPOLYGON SRID 4326.

При сохранении GeoJSON из Filament конвертировать его SQL-выражением:

ST_SetSRID(ST_Multi(ST_GeomFromGeoJSON(...)), 4326).

При чтении для админки использовать:

ST_AsGeoJSON(delivery_zones).

В Restaurant::deliveryZones() не пытаться напрямую JSON-декодить geometry.

Добавить проверку валидности:

ST_IsValid(delivery_zones);

ST_SRID(delivery_zones) = 4326;

geometry type = MULTIPOLYGON.

Вариант B — без PostGIS
Хранить GeoJSON как jsonb.

Проверку точки в полигоне делать в PHP.

Убрать зависимость от ST_Contains.

Для текущей архитектуры лучше Вариант A, потому что миграции уже создают PostGIS geometry. 

Этап 5. Заменить ST_Contains на более подходящую проверку
Для пользовательской доставки лучше использовать ST_Covers(delivery_zones, point) или ST_Intersects(delivery_zones, point).

Если зона может быть NULL, явно возвращать отдельную ошибку “зона ресторана не настроена”, а не просто “точка вне зоны”.

Логировать:

vendor id;

lat/lon;

есть ли зона;

валидна ли зона;

результат проверки.

Ожидаемый эффект: границы зоны перестанут давать ложный отказ.

Этап 6. Улучшить UX выбора точки
Добавить клик по карте для установки маркера, а не только drag marker.

После GPS/drag/select suggestion показывать координаты и источник: GPS / карта / подсказка / ручной ввод.

Если reverse geocode не нашёл адрес, не сбрасывать сценарий — оставить координаты и попросить ориентир.

Если точка вне зоны, показывать не только ошибку, но и понятное действие: “выберите другой ресторан” / “переместите точку ближе” / “введите другой адрес”.

Этап 7. Добавить диагностику и тесты
Минимальный набор проверок:

Feature-тест: ресторан без зоны → понятная ошибка “зона доставки не настроена”.

Feature-тест: точка внутри MULTIPOLYGON → заказ проходит.

Feature-тест: точка на границе → проходит, если выбран ST_Covers.

Feature-тест: точка вне зоны → отклоняется.

Unit-тест: GeoJSON из админки конвертируется в валидный PostGIS geometry.

Browser/Dusk-тест: при отсутствии ymaps пользователь видит ошибку, а не вечную загрузку.

Приоритетный порядок работ
Сначала проверить данные в БД: есть ли delivery_zones у ресторанов, какой тип, SRID, валидность.

Потом починить сохранение зон доставки в PostGIS-формат.

Затем заменить ST_Contains на ST_Covers или ST_Intersects.

После этого исправить состояние выбранной точки, чтобы дефолтный центр карты не считался адресом.

Потом унифицировать геокодинг: GEO_DRIVER=yandex + fallback.

В конце улучшить UX ошибок карты и геолокации.
