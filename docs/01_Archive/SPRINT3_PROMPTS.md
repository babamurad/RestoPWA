# Sprint 3 — ТЗ для AI-исполнителя

## 1) Контекст и цель спринта

**Проект:** RestoPWA (Laravel + PWA).

**Текущий/следующий приоритет Sprint 3:**
1. Реализовать и стабилизировать E2E smoke-тестирование ключевых пользовательских сценариев (сквозные тесты).
2. Выполнить прицельный багфикс по дефектам, выявленным E2E и ручной проверкой.
3. **Опционально:** добавить signed URL для отслеживания гостевых заказов (без авторизации).

**Бизнес-цель:**
- Снизить риск регрессий на критическом пути заказа.
- Сократить время проверки релиза.
- Повысить надежность сценария guest tracking.

---

## 2) Область работ (Scope)

### 2.1 Включено в Scope
- Подготовка инфраструктуры E2E smoke (инструмент: **Laravel Dusk**, структура, фикстуры, запуск локально/CI).
- Набор smoke-сценариев для критических пользовательских путей.
- Фикс дефектов уровня P0/P1, обнаруженных в ходе smoke.
- Добавление (опционально по флагу) signed URL механизма для гостевого трекинга заказа.
- Документация: как запускать, как дебажить, как интерпретировать падения.

### 2.2 Вне Scope
- Полный regression pack по всем edge-cases.
- Нагрузочные тесты.
- Рефакторинг вне контекста найденных дефектов.

---

## 3) Критические сценарии для E2E Smoke

Минимальный обязательный набор:

1. **Guest order flow (happy path)**
   - Открытие меню.
   - Добавление позиций в корзину.
   - Оформление заказа как гость.
   - Получение подтверждения.

2. **Order status visibility**
   - Переход на экран/ссылку статуса заказа.
   - Проверка отображения актуального статуса и ключевых полей.

3. **Basic resilience checks**
   - Поведение при пустой корзине.
   - Валидация обязательных полей checkout.

4. **(Опционально) Guest tracking via signed URL**
   - Генерация signed URL на этапе подтверждения.
   - Доступ к статусу только по корректной подписи.
   - Невалидная/просроченная подпись => корректный отказ (403/доменная ошибка).

---

## 4) Нефункциональные требования

- Smoke-тесты должны выполняться детерминированно (без флаки > 2%).
- **Обязательная очистка PWA кэша (Service Worker, IndexedDB) перед/между тестами для предотвращения flaky-тестов.**
- Время прогона smoke-пака в CI: целевое до 10 минут.
- Все тесты независимы, допускают изолированный запуск.
- Логи/артефакты падений: скриншоты + traces + video (если поддерживается инструментом).

---

## 5) Требования к багфиксу

- Приоритизация дефектов:
  - **P0:** блокирует заказ/оплату/статус — фикс в рамках текущего спринта обязателен.
  - **P1:** критичное ухудшение UX на ключевом пути — фикс обязателен, если не затрагивает высокий риск архитектурных изменений.
  - **P2:** допускается вынос в backlog с обоснованием.
- Каждый фикс сопровождается:
  - репродукцией;
  - root cause;
  - тестом (E2E или интеграционным), предотвращающим регрессию.

---

## 6) Опционал: signed URL для guest order tracking

### 6.1 Функциональные требования
- Для гостевого заказа формируется уникальная подписанная ссылка вида (используется встроенный `URL::temporarySignedRoute()`):
  - `/orders/{id}/track?signature=...&expires=...`
- Ссылка ограничена по времени (TTL).
- Проверка подписи выполняется на сервере.
- При невалидной подписи выдаётся 403 и безопасное сообщение без утечки чувствительных данных.

### 6.2 Безопасность
- Не использовать предсказуемые токены без подписи.
- Не логировать полную signed URL в открытые клиентские логи.
- Поддержать ротацию `APP_KEY` с documented последствиями (если применимо).

### 6.3 Критерии приемки
- Валидная подписанная ссылка открывает только соответствующий заказ.
- Изменение любого параметра ссылки ломает подпись.
- Просроченная ссылка недействительна.

## 7) Артефакты по итогам Sprint 3

1. E2E smoke suite в репозитории.
2. Отчет прогона (локально + CI): pass/fail + список дефектов.
3. Набор merged bugfix-коммитов.
4. Документация запуска и triage-инструкция.
5. (Если реализовано) документация по signed URL и ограничениям.

---

## 8) Definition of Done (DoD)

Спринт считается завершенным, когда:
- Все обязательные smoke-сценарии зеленые в CI.
- P0/P1 дефекты из smoke закрыты или формально согласованы к переносу.
- На каждый исправленный дефект есть автопроверка от регрессии.
- README/документация обновлены и актуальны.
- (Если включено в scope) signed URL покрыт тестами happy/negative path.

## 9) Промпты для AI (готовые шаблоны)

> Ниже промпты для поэтапной работы с AI-агентом. Используйте последовательно.

### Prompt 1 — Планирование Sprint 3

```text
Ты senior QA Automation + Tech Lead. Проект: Laravel PWA для ресторанных заказов.
Нужно сформировать технический план Sprint 3 с фокусом на E2E smoke + bugfix.
Контекст:
- Критический путь: guest order flow, checkout, order status tracking.
- Дополнительно: опциональный signed URL для guest tracking.
Требуется:
1) Разбить на workstreams (test infra, smoke scenarios, bugfix loop, optional signed URL).
2) Для каждого workstream дать задачи, риски, зависимости, оценку (S/M/L), критерии готовности.
3) Сформировать последовательность внедрения по дням (день 1..N).
4) Добавить exit criteria и rollback strategy.
Формат: Markdown-таблицы + четкий action list.
```

### Prompt 2 — Генерация E2E smoke test matrix

```text
Сгенерируй E2E smoke test matrix для Laravel PWA (гостевые заказы).
Нужно покрыть:
- Guest happy path checkout
- Валидации checkout
- Пустая корзина
- Отслеживание статуса заказа
- Негативные кейсы доступа к tracking
Для каждого теста укажи:
- ID, приоритет (P0/P1/P2), предусловия, шаги, ожидаемый результат,
- тип данных (фикстуры),
- что мокается/что не мокается,
- метку для CI smoke.
Отдельно выдели, какие тесты обязательны для merge gate.
```

### Prompt 3 — Реализация автотестов (кодогенерация)

```text
Ты senior QA engineer. Напиши production-grade E2E smoke tests для текущего проекта.
Требования:
- Используй существующий test stack проекта.
- Не ломай текущую архитектуру и линтеры.
- Добавь data-test селекторы только там, где это действительно нужно.
- Тесты должны быть изолированными и стабильными.
- Добавь before/after hooks и очистку данных.
- Подготовь краткую инструкцию запуска локально и в CI.
В ответе:
1) Покажи список создаваемых/изменяемых файлов.
2) Дай patch-ready код блоками по файлам.
3) Добавь команды запуска и отладки флаки.
```

### Prompt 4 — Bugfix loop по результатам smoke

```text
Ниже логи падений smoke-тестов (вставлю после этого сообщения).
Твоя задача:
1) Сгруппировать падения по root cause.
2) Отделить тестовые проблемы от продуктовых дефектов.
3) Для каждого продуктового дефекта предложить минимальный безопасный фикс.
4) Для каждого фикса предложить regression test.
5) Сформировать приоритетный план исправлений (P0/P1/P2).
Ограничения:
- Минимизировать риск побочных эффектов.
- Избегать большого рефакторинга в этом спринте.
Формат: таблица (дефект -> причина -> фикс -> тест -> риск -> приоритет).
```

### Prompt 5 — Опциональный signed URL дизайн

```text
Спроектируй signed URL для guest order tracking в Laravel.
Нужно:
- Endpoint и middleware-проверка подписи.
- TTL и политика истечения.
- Обработка невалидной/просроченной подписи.
- Требования безопасности (утечки, логирование, brute-force considerations).
- Набор тест-кейсов (happy + tamper + expired + wrong-order).
- План миграции без даунтайма.
Дай также пошаговый план внедрения с критериями приемки.
```

### Prompt 6 — AI-ревью PR перед merge

```text
Сделай строгий ревью изменений Sprint 3 как Staff Engineer:
Проверь:
- полноту smoke покрытия критического пути,
```
