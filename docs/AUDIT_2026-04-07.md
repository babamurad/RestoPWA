## 1. Краткий вывод (Executive Summary)

Проект уже находится выше уровня «сырого MVP»: есть рабочий пользовательский контур (каталог, корзина, checkout/success), офлайн-слой на Dexie, tenant-механика для API и существенный пласт тестов/документации.  
Основной риск сейчас не в «отсутствии архитектуры», а в **рассинхронизации требований, документации и фактических ограничений безопасности/операционного запуска**.

Если коротко:
- **Что хорошо:** ядро продукта собрано, есть направление развития (Q2 roadmap), есть offline-first фундамент.
- **Что опасно перед масштабированием:** безопасность трекинга заказа, единообразие auth/tenant policy для vendor-зоны, воспроизводимость quality gate в окружении.
- **Что нужно сделать первым:** финализировать security-policy, зафиксировать единый Technical Specification (ТЗ), и работать по спринтам только из этого источника.

## 2. Текущая зрелость по ключевым блокам

## 2.1 Product/Web Flow

**Факт:** основные пользовательские маршруты и auth-контур присутствуют:
- `/`, `/restaurants`, `/restaurants/{slug}`, `/cart`, `/checkout`, `/orders`, `/profile`,
- `/login`, `/register`, `/logout`,
- `/order/success/{id}`, `/order/{orderId}/track`.  

**Оценка:** `B` (функционально хорошо, но security-критерии для tracking нужно ужесточать).

## 2.2 API & Tenant Isolation

**Факт:**
- `api/v1` группа под `SetTenantContext`,
- есть `menu`, `restaurants`, `categories`, `orders`, `cart/sync`,
- пользовательские order endpoints под `auth:sanctum`.

**Оценка:** `B-` (базовая изоляция есть, но нужно довести до формализованной и проверяемой матрицы политик доступа).

## 2.3 Offline-first (PWA)

**Факт:** Dexie `CartService` реализует:
- корзину,
- totals,
- offline queue (`pendingOrders`),
- retry,
- bulk sync update.

**Оценка:** `B+` (логика заложена правильно; нужен стабильный CI-прогон frontend-тестов в целевом окружении).

## 2.4 Security & Access Control

**Факт:**
- `OrderSuccessController` проверяет user ownership,
- `OrderTrackingController` сейчас возвращает данные заказа по ID без owner/session-проверки,
- `docs/tenant-audit.md` ожидает `ensure.tenant + auth` для vendor-зоны,
- `routes/web.php` на vendor-группе пока только `ensure.tenant`.

**Оценка:** `C` (есть правильные элементы, но policy не доведена до однозначного и везде одинакового исполнения).

## 2.5 Документация и управление работами

**Факт:** есть несколько параллельных planning документов (`plan.md`, `pl.md`, roadmap, audit docs, DoD).

**Оценка:** `C+` (контента много, но слишком много источников — это создаёт перегруз и потерю фокуса).

## 3. Критические замечания (must-fix)

### P0-1: Доступ к трекингу заказа
`/order/{orderId}/track` и `/api/order/{orderId}/track` должны быть защищены: владелец заказа / временный signed token / гостевая сессионная привязка. Сейчас риск enumeration/утечки данных.

### P0-2: Единая политика доступа для vendor-контура
Нужно привести в соответствие документацию и роуты: если vendor-зона является служебной панелью, она должна быть под `auth + ensure.tenant` (и при необходимости role/permission).

### P0-3: Source of Truth
Нужен один главный документ (ТЗ) с приоритетами, критериями готовности и трекингом прогресса. Все остальные планы — ссылками на него.
