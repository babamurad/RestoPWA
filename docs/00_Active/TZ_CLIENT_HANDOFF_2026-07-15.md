# ТЗ: Доведение RestoPWA до состояния "готово к передаче заказчику"

**Дата:** 2026-07-15 (обновлено: решения заказчика от 2026-07-20)
**Статус:** согласовано по ключевым вопросам, детали SMS-канала — в работе
**Автор:** составлено на основе аудита текущего кода (`app/Domains/*`, `app/Filament/*`, `routes/*`, `database/migrations/*`)

## Зафиксированные решения заказчика (2026-07-20)
- **Бизнес-модель**: маркетплейс независимых ресторанов-партнёров → Модуль 8 (комиссия/выплаты) обязателен для запуска.
- **Оплата на старте**: только наличные курьеру. Онлайн-оплата картой переносится в бэклог после запуска (см. Модуль 1).
- **Языки интерфейса**: туркменский + русский (2 языка, без английского на старте).
- **Юридические страницы**: на старте достаточно страниц с примерным/шаблонным текстом (оферта, политика конфиденциальности); финальный юридически выверенный текст — отдельная задача заказчика позже.
- **Reverb (WebSocket)**: разворачивается силами заказчика — включено в критерии готовности инфраструктуры.
- **SMS-шлюз**: оператор в стране один — Altyn Asyr (бренд TMCell), гос. монополия (>90% рынка). Публичного самостоятельного SMS API для бизнеса не обнаружено (см. Модуль 4.5) — требуется прямые переговоры с оператором/Министерством связи Туркменистана. Технически закладываем архитектуру под подключение позже, без блокировки запуска.

## 0. Цель и рамки документа

Checkout/Order контур уже стабилизирован (см. `docs/00_Active/release_report.md`, `order_contract.md`). Это ТЗ описывает **всё остальное**, что необходимо, чтобы RestoPWA превратилась из работающего прототипа заказа еды в сервис, который можно передать заказчику как коммерческий продукт для ресторанов/кафе.

Каждый модуль ниже содержит: цель, модель данных, API-контракт, бизнес-правила, критерии приёмки. Это НЕ порядок реализации — приоритизация и фазы вынесены в раздел 15.

**Вне рамок этого документа:** доработки самого checkout-контракта (он уже заморожен в `order_contract.md`) и то, что уже реализовано (курьерский MVP, гео-зоны, офлайн-корзина, мультитенантность).

**Соглашения:**
- Все новые таблицы — `uuid` первичный ключ (`HasUuids`), как в существующих моделях (`Order`, `Restaurant`, `Product`).
- Все денежные поля — через `MoneyCast` (минимальные единицы/центы), как `Order::total`.
- Все новые API — под `routes/api.php`, префикс `v1`, `SetTenantContext` + `auth:sanctum` где нужен пользователь.
- Формат ошибок — единый с существующим (`success`, `reject_reason`/`errors`, `message`), см. `OrderRejectReason`.

## 1. Модуль: Оплата (Payments) — P0 (cash-only) / P2 (онлайн-эквайринг, бэклог)

### 1.1 Решение
Заказчик подтвердил: **на запуск — только наличные курьеру**. Онлайн-оплата картой физически не реализуется в этой итерации (нет доступного провайдера/договора с банком-эквайером в регионе на данный момент). Полноценная спецификация эквайринга (1.6–1.9 ниже) сохраняется в документе как **бэклог P2** — чтобы при появлении провайдера не пришлось перепроектировать `payment_status`/схему заново.

### 1.2 Что нужно сделать сейчас (P0, минимальный объём)
1. Убедиться, что `payment_method` в checkout жёстко ограничен значением `cash` (убрать/скрыть в UI варианты `card`/`sbp`, если они сейчас показываются как рабочие — иначе пользователь выберет нерабочий способ оплаты).
2. `Order.payment_status` должен корректно отражать cash-flow: `unpaid` → `paid` (проставляется вендором/курьером вручную по факту получения денег, например при переводе в `delivered`, либо отдельная кнопка "Оплата получена" в Filament/курьерском экране).
3. `UserPaymentMethod` (сохранённые "карты") — на текущем этапе не используется реально; либо скрыть раздел "Способы оплаты" в профиле клиента до появления online-оплаты, либо явно пометить как "в разработке", чтобы не создавать у пользователя ложное ожидание оплаты картой.

### 1.3 Критерии приёмки (P0)
- [ ] В checkout нельзя выбрать нерабочий способ оплаты (card/sbp скрыты или помечены "скоро").
- [ ] Есть явный, аудируемый способ отметить заказ оплаченным наличными (кто и когда отметил — фиксируется, например, через `OrderStatusHistory.metadata`).
- [ ] UI не вводит пользователя в заблуждение по поводу оплаты картой.

### 1.4–1.9 Бэклог P2: онлайн-оплата картой (реализовать, когда появится провайдер)

> Ниже — исходная спецификация, оставлена без изменений на будущее. Не входит в объём текущей передачи проекта.

#### 1.4 Функциональные требования
1. Онлайн-оплата картой через платёжный шлюз (эквайринг). Выбор провайдера — открытый вопрос (см. 1.9), архитектура должна быть провайдер-агностичной (адаптер).
2. Оплата инициируется после создания заказа (`status=pending`, `payment_status=awaiting_payment`) либо до (двухфазная модель — уточнить у провайдера при выборе).
3. Подтверждение оплаты приходит через **webhook** от платёжного провайдера — не полагаться на редирект клиента.
4. Поддержать возврат (refund) — полный и частичный, инициируется вендором/админом при отмене заказа с уже прошедшей оплатой.
5. Сохранённые карты (`UserPaymentMethod`) должны хранить **только токен провайдера**, не PAN/CVV (PCI DSS — процессинг данных карты никогда не должен касаться нашего бэкенда напрямую, только через SDK/redirect провайдера).

#### 1.5 Модель данных
Новая таблица `payment_transactions`:

| Поле | Тип | Описание |
|---|---|---|
| `id` | uuid | PK |
| `order_id` | uuid FK | → `orders.id` |
| `provider` | string | конкретный провайдер, когда определится |
| `provider_transaction_id` | string, nullable | ID транзакции у провайдера |
| `type` | enum | `payment`, `refund` |
| `status` | enum | `pending`, `succeeded`, `failed`, `cancelled` |
| `amount` | integer (money) | в минимальных единицах |
| `currency` | string(3) | `TMT` |
| `raw_payload` | json | сырой ответ провайдера (для дебага, без PII карты) |
| `idempotency_key` | string, unique | защита от дублей webhook |
| `created_at`/`updated_at` | timestamp | |

Изменения в `orders`:
- `payment_status` enum расширить: `unpaid`, `awaiting_payment`, `paid`, `refunded`, `partially_refunded`, `payment_failed` (сейчас — свободная строка, см. `Order::$fillable['payment_status']`). **Это расширение можно и нужно сделать уже сейчас** (в P0), даже без провайдера — cash-flow использует только `unpaid`/`paid`, остальные значения просто зарезервированы под будущее, без переделки схемы.

`UserPaymentMethod` — добавить поля `provider`, `provider_token` (шифруется через Laravel `encrypted` cast), убрать доверие к `card_mask` как источнику истины.

#### 1.6 API
```
POST   /api/v1/orders/{id}/payment/init        -> создаёт платёжную сессию, возвращает redirect_url/client_secret
POST   /api/v1/payment/webhook/{provider}      -> публичный, без auth, с проверкой подписи провайдера
GET    /api/v1/orders/{id}/payment/status       -> текущий статус оплаты (для поллинга фронта, если нет WS)
POST   /api/v1/vendor/orders/{id}/refund        -> инициирует возврат (роль: restaurateur/admin)
```

#### 1.7 Бизнес-правила
- Заказ не может перейти в `confirmed` вендором, если `payment_method != cash` и `payment_status != paid`.
- Webhook обязателен к идемпотентной обработке (по `idempotency_key`/`provider_transaction_id`) — повторный webhook не должен задваивать `paid`.
- Все суммы в `payment_transactions.amount` должны точно совпадать с `orders.total` на момент инициации — расхождение = `reject_reason: invalid_total`, платёж не создаётся.
- Логирование через `PIIMasker` (уже есть в проекте) — номера карт и токены никогда не пишутся в логи в открытом виде.

#### 1.8 Открытый вопрос (когда вернёмся к этому модулю)
- Какой платёжный провайдер/банк-эквайер будет доступен в регионе на тот момент — заново запросить у заказчика перед стартом реализации.
- Нужна ли рассрочка/сплит-оплата между рестораном и платформой в момент оплаты, или расчёт с вендором — отдельным циклом (см. Модуль 8)?

#### 1.9 Критерии приёмки (когда актуализируется)
- [ ] Оплата картой проходит end-to-end в sandbox-режиме провайдера.
- [ ] Webhook идемпотентен (тест: повторная отправка одного webhook не меняет состояние повторно).
- [ ] Возврат отражается в `payment_transactions` и статусе заказа.
- [ ] Ни один тест/лог не содержит PAN/CVV/полного токена карты.
- [ ] Feature-тесты: успешная оплата, отказ, webhook-дубликат, refund.

## 2. Модуль: Часы работы и доступность ресторана — P0

### 2.1 Проблема
В `restaurants` нет полей расписания. Клиент может оформить заказ в закрытый ресторан, вендор не может временно "поставить на паузу" приём заказов при перегрузке кухни.

### 2.2 Модель данных
Новая таблица `restaurant_schedules`:

| Поле | Тип | Описание |
|---|---|---|
| `id` | uuid | PK |
| `restaurant_id` | uuid FK | |
| `weekday` | tinyint (0-6) | 0=Пн |
| `opens_at` | time | |
| `closes_at` | time | |
| `is_closed` | boolean | выходной день |

В `restaurants` добавить:
- `is_paused` (boolean, default false) — ручная приостановка приёма заказов вендором ("не принимаем заказы сейчас").
- `pause_reason` (string, nullable).
- `timezone` (string, default из настроек платформы — важно, весь проект уже завязан на TM/993, см. `config/checkout.php`).

### 2.3 Бизнес-правила
- `GET /api/v1/restaurants/{vendor}` и список ресторанов должны отдавать вычисляемое поле `is_open_now: bool`.
- Создание заказа (`OrderPreconditionValidator`) должно добавить новую проверку: если ресторан закрыт (по расписанию или `is_paused`) → `reject_reason: RESTAURANT_CLOSED` (новый case в `OrderRejectReason`, 422).
- UI: на карточке ресторана и на странице меню — бейдж "Закрыто до HH:MM" / "Временно не принимает заказы", кнопка оформления заказа блокируется.
- Vendor-панель (Filament): переключатель "Пауза приёма заказов" на дашборде, редактор расписания по дням недели.

### 2.4 Критерии приёмки
- [ ] Заказ в закрытый ресторан отклоняется с понятным `reject_reason`.
- [ ] Вендор может поставить ресторан на паузу одним кликом из Filament, эффект применяется мгновенно (без деплоя/кэш-инвалидации вручную).
- [ ] Расписание корректно учитывает переход через полночь (напр. `22:00–02:00`).

## 3. Модуль: Автоназначение и логистика курьеров — P0/P1

### 3.1 Текущее состояние
Есть `Courier`, API `accept/status/location`, канал `courier.{id}`. Нет автоматического подбора курьера — заказ, видимо, либо назначается вручную вендором/админом, либо курьеры сами разбирают заказы из общего пула (нужно свериться с фактическим фронтом курьера — если такого экрана нет, это тоже P0 gap).

### 3.2 Функциональные требования
1. **Пул доступных заказов**: `GET /api/v1/courier/orders` (уже есть) должен отдавать заказы в статусе `ready_for_pickup` в радиусе N км от текущей позиции курьера (`current_lat/current_lon`), отсортированные по дистанции.
2. **Автоназначение (опционально, если заказчик хочет "Uber-like" авто-диспетчеризацию)**: при переходе заказа в `ready_for_pickup` — событие ищет ближайшего `available`-курьера этого вендора (или города, если курьеры общие для платформы — уточнить бизнес-модель) и отправляет push/WS-приглашение с таймаутом принятия (напр. 30 сек), при отказе/таймауте — следующему по списку.
3. **Статус курьера**: `Courier.status` уже есть поле — формализовать enum: `offline`, `available`, `busy`, `on_delivery`.
4. **Ограничение "одна доставка за раз"** (или "до N параллельных" — уточнить у заказчика) — курьер не может принять новый заказ, пока текущий не в терминальном статусе, если бизнес-модель это требует.
5. **Заработок курьера**: таблица `courier_earnings` (order_id, courier_id, amount, status: `pending`/`paid`), формируется при `status=delivered`. Базовая формула = фиксированная ставка за доставку + возможен процент от `delivery_fee` (сумма/правило — открытый вопрос к заказчику). При cash-only модели курьер, скорее всего, физически получает от клиента всю сумму заказа (товары + доставка) и должен часть отдать ресторану — это реальный операционный процесс, который нужно уточнить у заказчика: либо (а) курьер отдаёт ресторану стоимость товаров и оставляет себе `delivery_fee`, либо (б) деньги оседают у ресторана, а курьеру платит сам ресторан/платформа отдельно по факту доставки. От выбора зависит, нужен ли `courier_earnings` вообще на этом этапе, или расчёт с курьером — целиком внешний процесс ресторана.
6. **История и статистика курьера**: экран (или Filament-виджет) с количеством доставок/заработком за период.

### 3.3 API (дополнительно к существующим)
```
POST  /api/v1/courier/status              -> сменить available/offline (курьер вручную выходит на линию)
GET   /api/v1/courier/earnings            -> история начислений
POST  /api/v1/orders/{id}/assign          -> ручное назначение курьера (роль: restaurateur/admin)
```

### 3.4 Бизнес-правила
- Назначение курьера — событие `OrderAssigned` уже определено (`app/Events/OrderAssigned.php`) — использовать его как триггер push-уведомления курьеру.
- Курьер не должен видеть заказы другого вендора, если модель — "курьеры закреплены за рестораном" (сейчас `Courier.vendor_id` на это указывает). Если нужна общая биржа курьеров платформы — это отдельное архитектурное решение, зафиксировать ADR.

**Допущение, требующее подтверждения заказчиком (не удалось уточнить в диалоге — вопрос остался открытым):** поскольку подтверждена модель "маркетплейс независимых ресторанов", по умолчанию рекомендуется **общий пул курьеров платформы** (как в Wolt/Glovo/Яндекс.Еда) — независимые небольшие рестораны обычно не готовы содержать свою курьерскую службу, а платформа за счёт общего пула повышает утилизацию курьеров между заказами разных ресторанов. Текущая схема `Courier.vendor_id` (курьер жёстко привязан к одному ресторану) под эту модель не подходит и потребует миграции поля в nullable + добавления понятия "город/зона обслуживания" вместо жёсткой привязки к вендору. **Нужно явное подтверждение** этого допущения до начала разработки Модуля 3 — если у части ресторанов всё же будут свои курьеры, потребуется гибридная схема (см. критерии ниже).

### 3.5 Критерии приёмки
- [ ] Курьер видит только релевантные (по геозоне/вендору) доступные заказы.
- [ ] Повторное принятие уже принятого заказа другим курьером невозможно (race condition — покрыть тестом на конкурентный `accept`).
- [ ] Заработок корректно считается и виден курьеру после доставки.

## 4. Модуль: Уведомления — SMS и Email — P0 (email/push), SMS-канал — P1/внешне заблокирован

### 4.1 Проблема
Есть только web-push (VAPID), который недоступен для гостевых заказов, ненадёжен на слабом интернете и не работает, если пользователь закрыл вкладку/не установил PWA. Для рынка с невысоким проникновением стабильного интернета SMS — базовый канал доверия ("ваш заказ подтверждён").

### 4.2 Функциональные требования
1. Уведомления на ключевых переходах статуса: `confirmed` ("Заказ №X принят рестораном"), `delivering` ("Курьер выехал"), `delivered`, `cancelled` (с причиной) — отправляются по всем доступным на момент реализации каналам (на старте: push + email; SMS подключается отдельно, когда решится вопрос с оператором, см. 4.5).
2. Email — подтверждение заказа с чеком (детализация товаров/суммы) на email, если указан у пользователя; вендору — email/уведомление о новом заказе как fallback, если он не смотрит панель.
3. Настройка предпочтений пользователя: разрешить отключать не-критичные уведомления (маркетинг), но не критичные операционные (статус заказа).

### 4.3 Архитектура
- Использовать Laravel `Notification` с несколькими каналами: `mail`, `sms` (отдельный канал-адаптер — реализовать интерфейс так, чтобы провайдер подключался конфигурацией, без изменения бизнес-логики, когда появится доступ к шлюзу), `broadcast` (уже частично есть), `push` (обёртка над `PushNotificationService`).
- Подписаться на `OrderStatusUpdated` (уже существующее событие) единым `Listener`, который решает, какие каналы использовать в зависимости от наличия email/телефона/push-подписки у пользователя **и от того, какие каналы вообще включены** (`config('notifications.sms.enabled')` = false на старте — не ломается, просто SMS не уходит).
- Очереди (`queue`) обязательны для отправки — не блокировать HTTP-ответ на отправку SMS/email (см. также Модуль 10: production queue workers).

### 4.4 Критерии приёмки
- [ ] На каждый терминальный/ключевой переход статуса уходит хотя бы одно уведомление минимум по одному доступному каналу (push и/или email на старте).
- [ ] Отправка асинхронна (через queue), сбой одного канала не роняет обновление статуса заказа и не блокирует остальные каналы.
- [ ] PII (номер телефона, email) не попадает в общие логи в открытом виде (см. `PIIMasker`).
- [ ] Feature-тест: смена статуса → создана job в очереди на уведомление (mock).
- [ ] Архитектурно проверено: включение SMS-канала в будущем — это только конфигурация + один класс-адаптер провайдера, без изменений в `Listener`/бизнес-логике заказа.

### 4.5 SMS-канал: результат разведки по Altyn Asyr / TMCell

Туркменистан имеет фактически единственного мобильного оператора — **Altyn Asyr** (торговая марка **TMCell**), государственная монополия с долей рынка более 90%. Публичного самостоятельного (self-service) SMS API/личного кабинета для бизнес-рассылок у оператора в открытом доступе не обнаружено — это ожидаемо для гос. монополии со связным регулированием через Министерство связи Туркменистана: подключение A2P-рассылок (business-to-consumer SMS) обычно оформляется через прямой коммерческий договор с оператором, а не через самостоятельную регистрацию на сайте.

Международные SMS-агрегаторы (Messaggio, D7 Networks, sms.to, BudgetSMS, ExpertTexting, BulkSMS.com и аналогичные) заявляют номинальное покрытие Туркменистана через межоператорские SS7/SMPP-интерконнекты, но:
- фактическая доставляемость в гос. монопольную сеть закрытого рынка не гарантирована и требует тестовой проверки перед коммерческим использованием;
- вероятны дополнительные регуляторные ограничения на международный SMS-трафик, входящий как бизнес-рассылка.

**Рекомендация:**
1. Это не техническая, а коммерческо-регуляторная задача — требуется прямой контакт с Altyn Asyr/TMCell (или Министерством связи Туркменистана) по поводу условий подключения A2P SMS для юрлица. Разработка не может закрыть этот вопрос самостоятельно.
2. Параллельно можно протестировать 1-2 международных агрегатора на реальную доставляемость тестовых SMS на туркменские номера (+993) — дёшево и быстро даст ответ, работает ли этот путь вообще, пока идут переговоры с оператором напрямую.
3. До решения вопроса — push-уведомления и email остаются основным каналом связи с клиентом; в UI можно рассмотреть добавление номера WhatsApp/Telegram поддержки как неформальный резервный канал связи по заказу (не входит в объём этого ТЗ, только как идея для обсуждения с заказчиком).
4. Технически проект не блокируется: адаптер под SMS закладывается сейчас (4.3), реальный провайдер подключается отдельной небольшой задачей, когда появится ответ от оператора.

## 8. Модуль: Комиссия платформы и выплаты вендорам — P0 (маркетплейс независимых ресторанов — подтверждено)

### 8.1 Проблема
Платформа мультитенантна, но нет расчёта, сколько платформа берёт с каждого заказа и что должна выплатить ресторану/курьеру. Без этого модуля бизнес-модель площадки не работает юридически и финансово.

**Важное следствие решения "оплата только наличными" (см. Модуль 1):** деньги от клиента физически получает курьер/ресторан напрямую, минуя платформу. Значит направление расчёта — **не "платформа платит ресторану", а "ресторан должен платформе комиссию"** (инвойс/дебиторская задолженность). `vendor_settlements.status` в этой модели означает не "выплачено вендору", а "комиссия оплачена вендором платформе" (наличными/переводом вне системы, отмечается вручную админом). Это меняет смысл поля `net_payable` — по факту это "к оплате в пользу платформы", а не "к выплате вендору". Явно отразить это в UI Filament (подписи "Задолженность по комиссии", а не "Выплата"), чтобы не запутать администратора и владельцев ресторанов. Модель "платформа собирает деньги и сама выплачивает вендору за вычетом комиссии" становится актуальной только после появления online-оплаты (Модуль 1, P2).

### 8.2 Модель данных
В `restaurants` добавить `commission_percent` (decimal, per-vendor или из глобального конфига по умолчанию).

Новая таблица `vendor_settlements` (реестр расчётов):

| Поле | Тип |
|---|---|
| `id` | uuid |
| `restaurant_id` | uuid FK |
| `period_from`/`period_to` | date |
| `gross_amount` | money (сумма заказов) |
| `commission_amount` | money |
| `net_payable` | money (при cash-only = сумма комиссии к получению платформой с вендора) |
| `status` | `draft`/`approved`/`paid` (`paid` = комиссия получена платформой) |
| `paid_at` | timestamp, nullable |

**Открытый вопрос к заказчику (не блокирует разработку, но нужен до первого реального расчётного периода):** как физически ресторан платит платформе комиссию при cash-only модели — банковским переводом по инвойсу, наличными представителю платформы, удержанием из будущих расчётов? Это операционный процесс вне кода, но влияет на то, какие поля/статусы нужны в `vendor_settlements` (например, номер инвойса, способ получения).

### 8.3 Функциональные требования
1. При каждом `delivered`-заказе фиксировать `commission_amount = order.total * restaurant.commission_percent` (снапшот на момент заказа — не пересчитывать задним числом при смене тарифа).
2. Периодическая (напр. еженедельная) генерация `vendor_settlements` — Artisan-команда/scheduled job.
3. Экран в Filament (admin): реестр расчётов, ручная отметка "выплачено" (первая итерация — без автоматической банковской интеграции, это отдельная большая задача и зависит от банка заказчика).
4. Вендор в своей панели видит свою историю начислений/выплат (read-only).

### 8.4 Критерии приёмки
- [ ] Комиссия по каждому заказу фиксируется неизменно в момент доставки.
- [ ] Смена тарифа комиссии не переписывает прошлые начисления.
- [ ] Admin может закрыть период и пометить как оплаченный.

## 10. Модуль: Производственная инфраструктура (Production Readiness) — P0

7. **Rate limiting** для публичных API уже частично есть (`throttle:60,1`) — пересмотреть лимиты для `orders`, `geo/*`, `auth/*` отдельно (сейчас общий лимит на всю v1-группу, что позволяет одному эндпоинту "съесть" лимит соседних — точечные лимиты для чувствительных путей).

## 12. Модуль: Юридический контур и комплаенс — P0 (заглушка), финальный текст — бэклог заказчика

**Решение заказчика:** на этом этапе достаточно страниц с примерным/шаблонным текстом. Юридически выверенные формулировки заказчик подготовит сам позже — это НЕ блокирует техническую передачу проекта, но техническая инфраструктура под документы должна быть готова, чтобы заменить текст было делом одной правки, а не разработки с нуля.

### 12.1 Что нужно реализовать сейчас
1. Страницы: **Пользовательское соглашение**, **Политика конфиденциальности**, **Публичная оферта** (условия доставки/возврата), доступны из футера/профиля без авторизации. Содержимое — типовой шаблонный текст (плейсхолдер), явно готовый к замене (например, хранить как отдельные Markdown/Blade-файлы или записи в БД с версионированием — не хардкодить в общих компонентах, чтобы замена текста не требовала участия разработчика).
2. Согласие с условиями — чекбокс при регистрации, с фиксацией даты/версии документа, на которую согласился пользователь (поле `users.terms_accepted_at` + `terms_version`). Технически реализуется полностью независимо от того, финальный текст или шаблонный — версионирование нужно с первого дня, иначе при замене текста заказчиком позже не будет фиксации, кто на что согласился.
3. Политика хранения и удаления данных (право пользователя запросить удаление аккаунта/данных) — минимум ручной процесс через поддержку (`SupportTicket` уже есть, можно переиспользовать канал).
4. Страницы доступны на обоих языках (ru/tk) — можно с одинаковым шаблонным текстом на первом этапе, важна сама структура двуязычности (см. Модуль 11).

### 12.2 Отложено (бэклог заказчика, не входит в объём этой передачи)
- Юридически выверенный текст документов на предмет соответствия законодательству Туркменистана о персональных данных/дистанционной торговле.
- Финальное согласование текста оферты (условия доставки, возврата, ответственности сторон) с юристом заказчика.

### 12.3 Критерии приёмки
- [ ] Регистрация невозможна без явного согласия с текущей (пусть и шаблонной) версией документов.
- [ ] Юридические страницы отдаются на обоих поддерживаемых языках.
- [ ] Замена текста документа на финальный (когда он появится у заказчика) не требует участия разработчика — достаточно отредактировать контент через простой механизм (файл/CMS-запись), версия/дата принятия при этом корректно обновляется для новых согласий.

## 13. Модуль: Меню — модификаторы и опции (доводка) — P2

`Product.modifiers` уже существует как JSON-коллекция с `finalPrice`, суммирующим `modifiers.sum('price')`. Проверить и доформализовать:
- Группы модификаторов с правилами (`required`/`optional`, `min`/`max` выбор — напр. "выберите 1 соус из 3", "добавьте до 3 топпингов").
- Валидация на бэкенде при оформлении заказа: выбранные модификаторы должны существовать у товара и соответствовать правилам группы (сейчас, судя по `OrderItemDTO`, модификаторы, вероятно, принимаются как есть без строгой серверной проверки состава — уточнить и закрыть как security/consistency gap: клиент не должен мочь подставить произвольную цену модификатора).

### 13.1 Критерии приёмки
- [ ] Сервер пересчитывает цену модификаторов из БД, а не доверяет цене от клиента (аналогично существующей защите `invalid_price`/`invalid_total` на уровне товара).

## 14. Сквозные требования (все модули)

1. **Тесты обязательны**: по правилу проекта (см. `README.md`) — ни один PR по Order/Checkout/Payment не мержится без теста. Это правило распространяется на все новые модули из этого ТЗ, затрагивающие деньги/статус заказа (Payments, Promo, Settlements).
2. **Тестовые роли**: у каждого нового API-эндпоинта — явная проверка прав по `UserRole` (`admin`/`restaurateur`/`client`/`courier`), тест на 403 для чужой роли.
3. **PII-safe логирование**: новые логи проходят через `PIIMasker`, как уже сделано для гео-логов.
4. **Единый contract-style**: новые сущности документируются в `docs/00_Active/` по аналогии с `order_contract.md` — обновлять `order_contract.md`, если поля payload меняются (промокод, discount_amount).

## 16. Definition of Done — "готово к передаче заказчику"

Проект можно считать готовым к передаче, когда:

- [ ] Все P0-модули реализованы, покрыты тестами, прошли через тот же RC-процесс, что и checkout (`rc_checklist.md`).
- [ ] Полный цикл заказа проходим end-to-end на проде: клиент оформляет и **оплачивает** заказ → ресторан подтверждает (с учётом часов работы) → курьер назначается и **получает оплату за доставку** → клиент отслеживает в реальном времени по WebSocket → получает SMS/email на каждом статусе → может оставить отзыв.
- [ ] Юридические страницы и согласие с офертой на месте, ПДн обрабатываются в соответствии с локальным законодательством (подтверждено заказчиком/юристом).
- [ ] Комиссия платформы считается и есть реестр расчётов с ресторанами.
- [ ] Production-инфраструктура (WS, очереди, бэкапы, мониторинг, CI/CD) поднята и проверена, а не только описана в чек-листе.
- [ ] Есть базовая аналитика для вендора и платформы.
- [ ] Интерфейс локализован минимум на языки, согласованные с заказчиком.
- [ ] Передана эксплуатационная документация: `rc_checklist.md`, runbook инцидентов, доступы (без секретов в репозитории), схема инфраструктуры.
