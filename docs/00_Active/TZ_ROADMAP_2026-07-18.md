## 3. P0 — сделать в первую очередь (не зависит от внешних факторов)

### 3.1 Исправить баг несовпадения способов оплаты
Во фронтенде (`CheckoutPaymentStep.vue`) отправляются значения `cash`/`terminal`/`online`, а бэкенд (`OrderPreconditionValidator::validatePaymentMethod`) принимает только `cash`/`card`/`sbp`. Сейчас выбор "Картой при получении" или "Онлайн оплата" гарантированно ловит `INVALID_PAYMENT_METHOD`.
- [ ] Свести список значений к единому enum на фронте и бэке.
- [ ] Пока нет эквайринга — оставить реально рабочим только `cash` (наличные/карта курьеру физически, без онлайн-транзакции), остальные варианты либо скрыть в UI, либо явно пометить "скоро".
- [ ] Тест на `OrderPreconditionValidator` под актуальный список методов.

### 3.2 Server-side валидация цены модификаторов (безопасность)
Сейчас `OrderItemDTO` принимает цену модификаторов от клиента без сверки с каталогом — клиент теоретически может подделать сумму.
- [ ] При создании заказа пересчитывать цену модификаторов из `Product.modifiers` в БД, а не доверять `item.total_price` от клиента.
- [ ] Тест: заказ с подделанной ценой модификатора отклоняется / пересчитывается.

### 3.3 Реальный реалтайм-трекинг вместо polling
Backend (Reverb, события `OrderStatusUpdated`, `OrderAssigned`, `CourierLocationUpdated`, `routes/channels.php`) уже готов, но фронт не подключён.
- [ ] Поднять Reverb-процесс (для OSPanel — уточнить, поддерживает ли он долгоживущие процессы; если нет, Reverb можно гонять локально просто как `php artisan reverb:start` отдельным процессом, панель тут не блокер).
- [ ] Подключить `laravel-echo`/`@laravel/echo-vue` в `resources/js/bootstrap.js` (сейчас там только `window.axios`).
- [ ] Заменить/дополнить `setInterval(pollStatus, 5000)` в `resources/views/order/tracking.blade.php` на подписку через `Echo.private('order.{id}')`, оставить polling как fallback при обрыве WS.
- [ ] Проверить, что `BROADCAST_CONNECTION` не остаётся `log` в проде.

### 3.4 Автоназначение курьера (авто-диспетчеризация)
Сейчас только ручное назначение. Минимальная версия без сложного алгоритма:
- [ ] При переходе заказа в `ready_for_pickup` — искать ближайшего `available`-курьера (по вендору или по городу — зависит от бизнес-модели, см. вопрос ниже) и слать push/WS-приглашение с таймаутом на принятие (например, 30 сек), при отказе/таймауте — следующему.
- [ ] Формализовать `Courier.status` как enum: `offline`, `available`, `busy`, `on_delivery`.
- [ ] Тест на конкурентный `accept` — заказ не должен достаться двум курьерам одновременно.
- **Открытый вопрос:** курьеры закреплены за конкретным рестораном или это общий пул платформы? Без ответа автоназначение не спроектировать точно.

## 7. Сквозные правила (для всех новых модулей)

1. Ни один PR по деньгам/статусу заказа не мержится без теста — существующее правило проекта, распространить на промокоды, отзывы (влияют на рейтинг), settlements.
2. Каждый новый API-эндпоинт — явная проверка роли (`UserRole`) + тест на 403 для чужой роли.
3. Новые логи — через существующий `PIIMasker`, без PII в открытом виде.
4. Все денежные поля — через `MoneyCast`, как в `Order`.
5. При изменении payload заказа (промокод, discount_amount) — обновлять `docs/00_Active/order_contract.md`.
