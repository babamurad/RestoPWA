# План завершения проекта от 03.06.2026

## Что уже сделано

### 1. База проекта и стек
Проект заявлен как PWA-платформа доставки еды с offline-first корзиной, Service Worker, мультитенантностью через `X-Vendor-ID` и real-time трекингом заказов.

- Есть npm-скрипты для сборки и Vitest-тестов: `build`, `dev`, `test`.
- Есть Composer-скрипты для `setup`, `dev`, `test`, `lint` и `check`.

### 2. Checkout/order backend уже достаточно зрелый
- Backend принимает `POST /api/v1/orders` как публичный endpoint внутри v1-группы с tenant middleware.
- В [OrderController.php](file:///c:/OSPanel/home/RestoPWA/app/Domains/Order/Http/Controllers/Api/OrderController.php) есть нормализация адреса, товаров, суммы, customer-полей и проверка авторизации перед созданием заказа.
- Есть [OrderPreconditionValidator.php](file:///c:/OSPanel/home/RestoPWA/app/Domains/Order/Validators/OrderPreconditionValidator.php), который последовательно проверяет vendor, items, address, total, contacts, payment method и min order.
- Есть enum причин отказа заказа: empty cart, multi vendor, invalid total, outside delivery zone, invalid phone, unauthorized, server error и др.
- Идемпотентность частично реализована через заголовок `X-Idempotency-Key`: backend ищет уже созданный заказ пользователя с таким ключом и возвращает его как duplicate без создания дубля.

### 3. Frontend и offline-контур частично готовы
- Есть [OrderMapper.js](file:///c:/OSPanel/home/RestoPWA/resources/js/services/OrderMapper.js), который маппит клиентскую корзину в формат API: `product_id`, `product_name`, `quantity`, `unit_price`, `total_price`, `modifiers`, `image`.
- [OrderMapper.js](file:///c:/OSPanel/home/RestoPWA/resources/js/services/OrderMapper.js) также умеет собирать полный payload с `vendor_id`, `items`, `total`, `delivery_fee`, `payment_method`, `customer_name`, `customer_phone`, `address`, `idempotency_key`, `trace_id`.
- В [CheckoutPage.vue](file:///c:/OSPanel/home/RestoPWA/resources/js/vue/pages/CheckoutPage.vue) есть сборка payload, offline queue при сетевой ошибке и регистрация background sync.
- Service Worker ([sw.js](file:///c:/OSPanel/home/RestoPWA/public/sw.js)) умеет читать pendingOrders из IndexedDB, отправлять их в `/api/v1/orders`, удалять успешные, обрабатывать duplicate, 401/403 и постить сообщения клиентам.

### 4. Тестовый каркас есть
- JS unit-тесты покрывают [OrderMapper.test.js](file:///c:/OSPanel/home/RestoPWA/resources/js/services/OrderMapper.test.js), включая формат items, total, полный payload, idempotency key и trace id.
- Есть API-тесты reject reasons: empty cart, no vendor, vendor not found, inactive vendor, invalid price, invalid total и др.
- Есть API-тест идемпотентности заказа.
- Есть Dusk smoke-тесты для empty cart, multi-vendor CTA и unauthorized recovery.


## Главные проблемы, которые мешают считать проект завершённым

### P0 — frontend checkout payload не совпадает с backend-валидацией customer-полей
- В [CheckoutPage.vue](file:///c:/OSPanel/home/RestoPWA/resources/js/vue/pages/CheckoutPage.vue) телефон и имя кладутся внутрь `address.name` и `address.phone`, но не передаются как top-level `customer_name` и `customer_phone`.
- Backend же заполняет `customer_name`/`customer_phone` из top-level request-полей или из авторизованного пользователя. Если у пользователя нет телефона в профиле, `customer_phone` станет пустой строкой.
- После этого Laravel-валидация требует `customer_name` и `customer_phone` как top-level поля.
- **Что сделать:** в [CheckoutPage.vue](file:///c:/OSPanel/home/RestoPWA/resources/js/vue/pages/CheckoutPage.vue) перестать вручную собирать контракт и использовать `buildOrderPayload()` из [OrderMapper.js](file:///c:/OSPanel/home/RestoPWA/resources/js/services/OrderMapper.js), либо минимум добавить top-level `customer_name` и `customer_phone` в payload.

### P0 — trace_id заявлен как end-to-end, но Vue checkout фактически его не отправляет
- Контракт требует `trace_id`, а документация говорит, что он должен генерироваться на frontend и передаваться во всех API-запросах.
- Но [orders.js](file:///c:/OSPanel/home/RestoPWA/resources/js/vue/api/orders.js) отправляет только `X-Vendor-ID` и `X-Idempotency-Key`, без `X-Trace-Id`.
- Backend берёт trace id из `X-Trace-Id` или генерирует новый на сервере. Это значит, что сейчас trace не end-to-end, а в большинстве Vue checkout-сценариев начинается уже на backend.
- **Что сделать:** генерировать trace id при входе в checkout, хранить его в состоянии checkout/order payload, отправлять и в body `trace_id`, и в header `X-Trace-Id`.

### P0 — документация контракта заказа устарела и противоречит коду
- Документ [order_contract.md](file:///c:/OSPanel/home/RestoPWA/docs/00_Active/order_contract.md) описывает payload с `items[].id`, `items[].price`, `total_price`, `delivery_address`, `coordinates.lat/lng` и payment method `cash` | `card` | `online`.
- Реальный JS mapper и backend DTO используют другой формат: `product_id`, `unit_price`, `total_price` на item, общий `total`, `delivery_fee`, объект `address.lat/lon`, `customer_name`, `customer_phone`.
- Backend validator также разрешает payment methods `card`, `cash`, `sbp`, а не `online`.
- **Что сделать:** обновить [order_contract.md](file:///c:/OSPanel/home/RestoPWA/docs/00_Active/order_contract.md) под фактический контракт и сделать этот документ действительно single source of truth.

### P0 — release-документы противоречат друг другу

- Но [release_dod.md](file:///c:/OSPanel/home/RestoPWA/docs/00_Active/release_dod.md) содержит незакрытые чекбоксы по happy path, top errors, offline queue, контракту, идемпотентности, performance, PII-safe logging, trace, monitoring, rollback и backup.

### P1 — offline/auth recovery не выглядит полностью замкнутым
- Service Worker при 401/403 отправляет клиентам сообщение `ORDER_AUTH_REQUIRED`.
- Но основной listener в [app.js](file:///c:/OSPanel/home/RestoPWA/resources/js/app.js) сейчас обрабатывает только `ORDER_SYNCED`.
- Есть отдельный блок в SW, который транслирует `ORDER_AUTH_REQUIRED` в `auth-required-from-sw`, но в просмотренном клиентском коде нет явного полного UX-потока восстановления: показать логин, вернуться, повторить pending order.
