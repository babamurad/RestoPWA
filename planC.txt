Phase C — Contacts, Localization & Validation Hardening Implementation Plan
Цель
Сделать оформление заказа устойчивым для разных регионов и сценариев ввода контактов: убрать жёсткую привязку к +993, синхронизировать frontend/backend валидацию телефона, улучшить UX ошибок и снизить ложные отказы оформления.

User Review Required
IMPORTANT
В Phase C поведение меняется: вместо принудительной нормализации к +993... ввод телефона станет конфигурируемым по региону (или универсальным E.164), и часть “автоматических исправлений” будет заменена на явные подсказки пользователю.
Это может изменить привычный flow для текущих пользователей TM, но уменьшит число неочевидных ошибок и отклонений заказа.

Risks & Mitigations
Риск: пользователи с привычным локальным форматом путаются.
Mitigation: понятные примеры/placeholder + автонормализация безопасных кейсов.

Риск: расхождение Livewire/API валидации.
Mitigation: единый validation policy (shared rule builder или дублирование по одному конфигу).

Риск: оффлайн-заказы с устаревшими контактами не синкаются.
Mitigation: explicit needs_user_action и UX-подсказка.

Риск: слишком подробные логи с PII.
Mitigation: маскирование телефона/имени в логах, хранить только reason/codes.

Manual Verification
Strict mode: номер вне региона блокируется корректным сообщением.

E.164 mode: международные номера принимаются.

Offline queued order: при невалидных контактах не синкается молча, отображается требование действия.

Telemetry: в логах есть traceId + reason, без утечки полного телефона.

Definition of Done (DoD)
 Контактная валидация работает одинаково в Livewire и API.

 Жёсткая привязка к +993 убрана/сделана конфигурируемой.

 UX ошибок понятен и локализован.

 В логах есть причины отказа и trace correlation.

 Есть автоматические тесты на policy modes и reject reasons.

 Нет утечки PII в логи.
